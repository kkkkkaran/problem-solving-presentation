- This slide introduces the three-step methodology that forms the core of my talk - The steps build on each other in a logical progression - Visualize: First, you need to clearly define what success looks like - Work backwards: Then identify the path from end goal back to where you are now - Break it down: Finally, divide the journey into manageable pieces - The steps-to-success image reinforces the methodical, progressive nature of this approach - This is a general framework that can be applied to any complex problem - I'll be showing concrete examples of how we applied this to real coding challenges

Step 1 - Visualize the End Goal

Starting without a clear vision is like building IKEA furniture without the final picture
  • What problem are we really solving?
  • What does success actually look like?
  • Who benefits and how?
"If you don't know where you're going, any road will get you there."

Real Example - Visualizing the Goal

Warehouse Dispatch System at Aussie Broadband

Developer Experience Hell:

  • Hardcoded products requiring full deployment cycles
  • No testing possible - unit, feature, or local
  • 1+ hour CI pipeline for any change
  • Session-based spaghetti code with extreme cognitive load
  • Key personnel risk - "you touch it, you own it"

Business Impact:

  • New products needed developer involvement
  • Warehouse operations stalling on edge cases
  • Ineffective order prioritization
  • Hardware-locked dispatch kiosks
  • Complex location expansion process
It was like playing Jenga with a production system
- The DX problems created a vicious cycle: high risk meant only experts could change the code - Business impact was multiplicative: each new product or location amplified existing problems - This was more than just technical debt - it was actively holding back business growth

The Vision

Breaking Free from Legacy Constraints

Technical Goals:

  • Configuration-driven product management
  • Comprehensive testing suite
  • Independent service deployments
  • Clean, modular architecture

Business Outcomes:

  • Self-service product management
  • Graceful failure handling
  • Smart order prioritization
  • Browser-based dispatch system
  • Easy warehouse expansion
"Instead of just 'making it work with PHP 8', we saw an opportunity to transform warehouse operations."
Our first attempt looked more like a Rube Goldberg machine, but at least it had tests!
- Vision focused on breaking these cycles through automation and configuration - Key insight: Moving from "developer-dependent" to "developer-enabled" system - Each technical goal directly enabled a business outcome - The transformation wasn't just technical - it changed how the business could operate

Breaking down the monolith allowed us to:

  • Isolate failures to prevent queue stalls
  • Implement order prioritization
  • Create a device-agnostic solution
  • Add new warehouses via simple DB updates
Key components we identified: - Core job management: Handling creation, validation, and lifecycle - Hardware & device management: Making it device-agnostic - Carrier integration system: Abstracting shipping providers - Boxing & label generation: Handling physical dispatch needs - User authentication: Managing access control - Warehouse allocation: Smart routing of orders - Item scanning/validation: Quality control - Event handling with observers: System-wide notifications Each component was further broken down into: - Controller layer: Handling web requests cleanly - Service layer: Encapsulating business logic - Model layer: Managing data structure - Interface-based design: Especially important for shipping carriers This modular approach meant: - Components could be developed and tested independently - New features could be added without touching existing code - Teams could work in parallel on different parts - Each piece had a single, clear responsibility

Why This Works

Team Benefits:

  • Clearer communication
  • More accurate estimates
  • Easier delegation
  • Start with small, achievable goals

Technical Benefits:

  • Faster identification of roadblocks
  • Higher quality solutions
  • More maintainable code
  • Testable components

For our dispatch system, this approach gave us:

  • Specialized controllers for each aspect (Jobs, Devices, BoxSizes)
  • Clean model relationships with clear responsibilities
  • Modular services that could be independently tested
  • Clear interfaces between components
"Our dispatch system achieved clean separation of concerns with services that follow SOLID principles."
How to get started with this approach: - Take 15 minutes to write out what success looks like - Identify your 3 most critical integrations/dependencies - Break the problem into logical controllers, models and services Real examples from our dispatch system: - Started with core job management - Added carrier integration one provider at a time - Built device management as a separate service - Gradually moved features from monolith to new services Key to success: - Start with small, achievable components - Build incrementally - Test each piece thoroughly - Keep services focused and single-purpose Remember: This approach works because it makes complex problems manageable through systematic breakdown and clear boundaries between components.
