This slide introduces the three-step methodology that forms the core of my talk

The steps build on each other in a logical progression:
- Visualize: First, you need to clearly define what success looks like
- Work backwards: Then identify the path from end goal back to where you are now
- Break it down: Finally, divide the journey into manageable pieces

Step 1 - Visualize the End Goal

Starting without a clear vision is like building IKEA furniture without the final picture
  • What problem are we really solving?
  • What does success actually look like?
  • Who benefits and how?
"If you don't know where you're going, any road will get you there."
[Rhetorical question to audience] "How many of you have tried to build IKEA furniture without looking at the final picture? How did that work out?"

The IKEA instruction meme illustrates why having a clear vision is so important:

When we skip visualization, we often build the wrong solution:
- Building features no one will use
- Solving symptoms instead of root causes
- Creating technical debt through rushed implementations

Good questions to ask at this stage:
- What would a perfect solution look like if we had unlimited resources?
- How will we know when we've succeeded?
- Who are all the stakeholders and what do they need?

This stage might feel like "wasting time" but it actually saves time overall

Real Example - Visualizing the Goal

Warehouse Dispatch System at Aussie Broadband

Developer Experience Hell:

  • No testing possible - unit, feature, or local
  • 1+ hour CI pipeline for any change
  • Session-based spaghetti code with extreme cognitive load
  • Key personnel risk - "you touch it, you own it"

Business Impact:

  • New products needed developer involvement
  • Warehouse operations stalling on edge cases
  • Ineffective order prioritization
  • Hardware-locked dispatch kiosks
  • Complex location expansion process
It was like playing Jenga with a production system
Introduction to the dispatch system:
- This was a critical system at Aussie Broadband used for warehouse order processing
- It handled everything from order creation to shipping label generation
- Used by warehouse staff to fulfill customer orders for modems, routers, and other hardware
- Originally built as a monolithic PHP application with jQuery for front-end interactions
- Had been in production for 10+ years with minimal architectural changes

[What problem are we really solving?]
- The initial request was simple: "Make it work with PHP 8"
- But visualizing the goal helped us see the real problems beneath the surface
- This wasn't just about PHP compatibility - it was about breaking free from fundamental limitations
- By asking "what problem are we really solving?" we uncovered these deeper issues:

- Even simple product updates needed a full deployment
- Product catalog couldn't be managed by business users

[Fragment: No testing possible - unit, feature, or local]
- Code was untestable due to tight coupling
- No separation between business logic and presentation
- Developers had to test manually in production

[Fragment: 1+ hour CI pipeline for any change]
- Simple fixes took an entire day to deploy
- Created bottlenecks in development workflow
- Made quick fixes nearly impossible

[Fragment: Session-based spaghetti code with extreme cognitive load]

- Code flow was difficult to follow and understand
- High mental overhead to make even small changes

- Knowledge wasn't transferable to new team members
- Created bottlenecks and single points of failure

[Fragment: New products needed developer involvement]

- Product launches delayed waiting for developer availability
- Competitive disadvantage in fast-moving market

The Vision

Breaking Free from Legacy Constraints

Technical Goals:

  • Configuration-driven product management
  • Comprehensive testing suite
  • Independent service deployments
  • Clean, modular architecture

Business Outcomes:

  • Self-service product management
  • Graceful failure handling
  • Smart order prioritization
  • Browser-based dispatch system
  • Easy warehouse expansion
"Instead of just 'making it work with PHP 8', we saw an opportunity to transform warehouse operations."
Our first attempt looked more like a Rube Goldberg machine, but at least it had tests!
By visualizing the end goal, we transformed a PHP 8 upgrade into an opportunity for real change.

For developers, we envisioned:
- Configuration-driven system: Change products without code deployments
- Comprehensive testing: Enable confident, rapid development cycles
- Independent services: Deploy components separately, shorter feedback loops
- Clean architecture: Components with clear responsibilities and boundaries

For the business, this meant:
- Self-service: Product teams could manage their own offerings
- Resilience: One issue wouldn't bring down the entire system
- Smart prioritization: Ability to expedite urgent orders when needed
- Browser-based system: Works on any device, no proprietary hardware
- Simple expansion: New warehouses added through configuration

This vision became our North Star - whenever we faced tough decisions, we'd ask: "Does this align with breaking free from legacy constraints?"

Step 2 - Work Backwards

Find your starting point

  • Identify critical requirements
  • Map dependencies
  • Find the shortest path from end to start
"Begin with the end in mind, then figure out how to get there."
GPS manufacturers missed their chance with the "figure out where you're going first" button
GPS meme connection: Just like a GPS needs both destination AND current location, problem-solving requires working backwards. Why working backwards works: * Reveals dependencies you'd miss when planning forward * Prevents getting stuck in implementation details too early * Creates a clearer roadmap by focusing on outcomes first Identify critical requirements: * What must absolutely be true for your solution to work? * What are the non-negotiable requirements? * Example: "Users must be able to prioritize urgent orders" Map dependencies: * What systems will your solution need to interact with? * What data flows are necessary between components? * Example: "We'll need shipping carrier APIs for labels" Find the shortest path: * What's the minimum viable solution? * Which approach gives maximum impact with minimal change? * Example: "We could abstract carrier logic without changing UI first" Working backwards feels unnatural at first, but it's like planning a road trip by starting with your destination and working back to your driveway.

Step 3 - Break It Down

Create achievable components

  • Divide by function
  • Create logical groupings
  • Keep components manageable (1-2 day tasks)
"The best way to eat an elephant is one bite at a time."

Key components we identified:
- Core job management: Handling creation, validation, and lifecycle
- Hardware & device management: Making it device-agnostic
- Carrier integration system: Abstracting shipping providers
- UI layer: Providing a flexible frontend

This modular approach meant:
- Components could be developed and tested independently
- New features could be added without touching existing code
- Teams could work in parallel on different parts
- Each piece had a single, clear responsibility

Breaking down the monolith allowed us to:
- Isolate failures to prevent queue stalls
- Implement order prioritization
- Create a device-agnostic solution
- Add new warehouses via simple DB updates

  • More accurate estimates
  • Easier delegation
  • Start with small, achievable goals

Technical Benefits:

  • Faster identification of roadblocks
  • Higher quality solutions
  • More maintainable code
  • Testable components

For our dispatch system, this approach gave us:

  • Specialized controllers for each aspect (Jobs, Devices, BoxSizes)
  • Clean model relationships with clear responsibilities
  • Modular services that could be independently tested
  • Clear interfaces between components

Technical benefits compound over time:
